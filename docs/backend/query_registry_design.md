# Query Registry Design (Traceable + Scalable)

## Goal
Replace opaque answer generation paths with a traceable analytics execution model that:
- Preserves SQL guardrails.
- Scales to many intents without hardcoded capabilities lists.
- Keeps agent outputs auditable (which query, params, SQL, evidence).

## Current Pain Points
- `answer_business_question_tool` can return final business answers without exposing a unified SQL trace contract.
- Deterministic intent detection covers many intents, but executable SQL handlers cover only a subset.
- Capability messaging is manually hardcoded instead of derived from executable coverage.

## Proposed Architecture

### 1) Query Registry As Source of Truth
Create a registry artifact (YAML/JSON) with one record per executable capability.

Suggested path:
- `src/agent/semantics/query_registry.yaml`

Suggested schema per entry:
- `query_key`: stable identifier (e.g., `income_period_week`)
- `intent_id`: semantic intent id (must exist in `intents.yaml`)
- `status`: `enabled | disabled | preview`
- `description_es`: short human-readable description
- `required_slots`: e.g. `merchant_id`, `time_range`
- `optional_slots`: e.g. `top_n`, `seller_id`
- `chart_allowed`: bool
- `capability_tags`: list for capability discovery (`ingresos`, `clientes`, `vendedores`)
- `sql_template`: parameterized SQL (or `query_builder_id` if generated by function)
- `response_schema`: expected output fields used for answer rendering
- `owner`: optional (team/person)
- `version`: optional semantic version

Registry-level metadata:
- `unsupported_topics` (single place for policy)
- `defaults` (currency/timezone/week definition)

### 2) Single Traceable Execution Tool
Introduce one primary analytics tool contract (name example):
- `run_analytics_query_tool(intent_id, slots, merchant_id=None, max_rows=20)`

Execution flow:
1. Validate `intent_id` + required slots against registry.
2. Resolve template/query builder and bind params.
3. Run through guarded SQL executor.
4. Return a strict trace payload.

Response contract (JSON) should always include:
- `ok`
- `intent_id`
- `query_key`
- `merchant_id`
- `resolved_sql` (or redacted_sql + fingerprint if needed)
- `params`
- `columns`
- `rows`
- `row_count`
- `facts_payload` (normalized facts)
- `chart_allowed`
- `errors` (if any)

This makes every factual answer reproducible and inspectable.

### 3) Keep Free-SQL Tool As Controlled Fallback
Keep `run_sql_analytics_tool` for long-tail exploration, but position it as secondary path.

Policy:
- First attempt: registry-backed `intent_id + slots` query.
- Fallback only if no registered query can satisfy the request.
- Fallback outputs must still emit trace payload with SQL and validation result.

### 4) Dynamic Capabilities (No Hardcoded Lists)
`assistant_capabilities_tool` should be generated from the registry:
- `supported`: entries where `status=enabled` grouped by tags.
- `preview`: entries where `status=preview`.
- `unsupported`: from registry policy section.

This removes drift between docs/prompts and real execution coverage.

### 5) Evaluations Alignment
Evals should validate traceability, not only answer text.

Add criteria:
- Query provenance present (`query_key`, `intent_id`, `resolved_sql|fingerprint`).
- Correct query family for expected intent.
- Evidence fields populated and consistent.

Update eval trajectory expectations from hardcoded tool names to:
- primary expected tool category: `registry_query_tool`
- optional fallback category: `sql_fallback_tool`

## Migration Plan

### Phase 0 (Preparation)
- Add registry file with entries for already-working intents:
  - `income_period`
  - `income_distribution`
  - `income_vs_previous`
- Add entries for currently failing intents as `disabled` or `preview`.

### Phase 1 (Traceability Foundation)
- Implement primary registry-backed tool contract.
- Keep existing tools wired, but route supported intents through registry tool first.
- Start emitting standardized trace payload.

### Phase 2 (Coverage Expansion)
- Migrate missing intents one by one to enabled registry entries:
  - `best_worst_day`
  - `customer_new_vs_returning`
  - `top_customers`
  - `inactive_customers`
  - `peak_hours`
  - `seller_performance`

### Phase 3 (Cleanup)
- Deprecate `answer_business_question_tool` once all enabled intents are registry-backed.
- Keep only:
  - registry query tool (primary)
  - safeguarded free-SQL tool (fallback)
  - dynamic capabilities tool

## Guardrail Notes
- Reuse existing guarded SQL validator/executor; do not bypass it.
- Keep merchant isolation requirement mandatory.
- Consider SQL fingerprint logging for safer traceability in production logs.

## Acceptance Criteria
- Every successful factual answer references a query provenance object.
- Capabilities output is auto-derived from registry status.
- No hardcoded supported-intent list in tool layer.
- Full eval report includes provenance checks and improved quality pass rate.
